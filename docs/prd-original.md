# 可信 AI 知识工作台产品开发文档

## 1. 项目概述

### 1.1 项目名称

可信 AI 知识工作台

可暂定英文名：Trustworthy AI Knowledge Workbench，简称 TAIKW。

### 1.2 项目定位

本系统定位为一个 AI 知识工作台，核心目标是为研究人员、企业团队、法律合规团队、知识管理团队、教育团队等用户提供一个“可上传资料、可调用模型、可运行本地模型、可追溯引用、可复核输出”的可信 AI 工作环境。

系统重点解决大语言模型在严肃知识工作中的核心痛点：模型输出可能存在幻觉、误引、过度推断、来源不明、难以复核等问题。产品通过资料库管理、文档结构化、检索增强生成、引用溯源、一键跳转原文、人工复核记录等机制，将模型输出从“不可验证的自然语言回答”转化为“有来源、有证据、可追踪、可复核的知识工作结果”。

### 1.3 产品愿景

构建一个通用、开源、可扩展、可私有化部署的可信 AI 知识工作台，让用户在使用大语言模型处理专业资料时，能够始终回到原始证据，降低模型幻觉对研究、决策和业务流程的影响。

### 1.4 核心价值

1. 降低模型幻觉风险：模型回答必须尽可能关联原文证据。
2. 提高专业资料处理效率：统一管理 PDF、DOCX、Markdown、TXT 等文档。
3. 支持多模型接入：用户可自定义 API Key、Base URL 和模型名称。
4. 支持本地模型运行：可导入 Hugging Face 模型权重或本地微调模型。
5. 支持人机协同复核：模型输出后，用户可一键跳转原文并记录复核意见。
6. 支持多端同步：Windows、Linux、Android 等端统一访问同一工作区。
7. 支持通用场景扩展：通过插件、标签体系、提示词模板和分析模板适配不同领域。
8. 适合开源生态：模块边界清晰，便于社区贡献文档处理器、模型适配器、主题模板和前端组件。

## 2. 产品目标

### 2.1 第一阶段目标：可信知识库 MVP

完成基础可用系统，支持用户上传资料、转换为结构化文本、构建索引、调用外部模型 API、生成带引用来源的回答，并支持点击引用跳转原文。

### 2.2 第二阶段目标：多端同步与复核闭环

支持 Windows、Linux、Android 多端使用；支持复核状态、批注、引用质量评估、回答修订记录，让模型输出进入可审计流程。

### 2.3 第三阶段目标：本地模型与开放生态

支持导入 Hugging Face 模型、本地微调模型和 OpenAI-compatible 本地推理服务；形成开放的模型适配器、文档解析器、任务模板、前端组件体系。

### 2.4 第四阶段目标：开源发布与知识工作平台化

形成稳定的开源代码仓库、模块化架构、插件机制、部署文档、示例数据集、主题模板和开发者贡献规范，使系统可以被个人、团队或企业二次开发。

## 3. 产品边界

### 3.1 系统要做什么

系统需要提供：

1. 多端应用安装与同步。
2. 用户、工作区、项目空间管理。
3. 文档上传、解析、转换、归档。
4. Markdown 作为统一中间格式。
5. 文档分块、全文检索、向量检索、混合检索。
6. 外部大模型 API 接入。
7. 本地模型导入和运行。
8. 基于资料库的问答、摘要、翻译、解释、比较。
9. 模型输出引用来源。
10. 一键跳转原文。
11. 用户复核、批注、修订和审计记录。
12. 模块化前端设计系统。
13. 插件化扩展能力。
14. 开源部署与二次开发能力。

### 3.2 系统不直接承诺什么

系统不应承诺完全消除大模型幻觉。更准确的产品表述应为：通过证据约束、引用溯源、人工复核和日志记录，显著降低幻觉风险，并提高错误发现与纠正效率。

系统也不应将模型输出直接定义为最终研究结论、法律意见、医疗意见、投资建议或企业决策结论。系统提供的是知识辅助、资料定位、文本处理和复核工具。

## 4. 核心需求

### 4.1 用户权限管理

系统应支持基础的用户权限管理功能，MVP 阶段采用两级权限模型：管理员与普通用户。

管理员拥有系统内的全部管理权限，主要包括：

1. 创建、删除和管理工作区。
2. 邀请、移除和管理用户。
3. 配置外部模型 API Key、Base URL、模型名称和调用参数。
4. 导入、加载、卸载和管理本地模型。
5. 管理 Hugging Face 模型权重、本地微调模型和模型运行后端。
6. 管理文档处理规则、索引规则、提示词模板和领域模板。
7. 查看系统日志、模型调用记录、文件处理记录和复核统计。
8. 删除或恢复资料、对话记录、引用记录和复核记录。
9. 修改系统级设置与工作区级设置。
10. 管理用户角色和工作区访问权限。
11. 配置是否允许普通用户上传资料、创建标签、导出结果或调用特定模型。

普通用户只拥有模型使用和研究协作相关权限，主要包括：

1. 查看其有权限访问的工作区和资料库。
2. 上传、查看和检索资料文件，具体是否允许上传可由管理员配置。
3. 使用管理员已配置好的模型进行问答、摘要、翻译、解释、比较等操作。
4. 查看模型输出中的引用来源。
5. 点击引用跳转至原始文件对应位置。
6. 对模型输出进行复核标记，例如可信、引用不足、存在误译、疑似幻觉等。
7. 添加批注、标签和个人备注。
8. 查看本人创建的对话记录、批注记录和复核记录。
9. 修改自己的基础账号信息，例如显示名称和密码。

普通用户不得执行以下操作：

1. 查看或修改 API Key。
2. 新增、删除或修改模型配置。
3. 导入、加载或卸载本地模型。
4. 修改系统级提示词模板。
5. 修改文档处理规则、索引规则或模型运行后端配置。
6. 删除其他用户上传的资料，除非管理员额外授权。
7. 删除其他用户的复核记录或批注。
8. 修改系统权限、工作区权限和用户角色。
9. 查看敏感系统日志、密钥配置或模型调用密钥。
10. 修改系统级设置或工作区级安全设置。

权限控制应在后端统一实现，前端只负责根据当前用户角色隐藏或禁用相关入口。任何关键权限判断不得只依赖前端。后端 API 应在每次请求时校验用户身份、所属工作区和角色权限，防止普通用户通过直接调用接口绕过限制。

MVP 阶段可采用简单的 RBAC 权限模型，即 Role-Based Access Control。系统预设两个角色：admin 和 user。后续如需要支持企业场景，可扩展为 owner、admin、maintainer、researcher、reviewer、readonly 等更细粒度角色。

### 4.2 多端同步

系统应支持作为应用程序安装在以下平台：

1. Windows。
2. Linux。
3. Android。

推荐使用 Flutter 或类似跨平台框架实现统一代码基础。桌面端和移动端通过统一后端 API 访问数据。所有端应共享账号、工作区、资料库、对话记录、引用记录、批注记录和模型配置。

多端同步应采用“后端服务 + 数据库 + 对象存储”的中心化同步方式。客户端可以缓存最近使用的资料和对话，但最终数据以服务器端为准。

### 4.3 自定义 API Key 与 URL

系统应允许用户或管理员配置外部模型服务：

1. API Key。
2. Base URL。
3. 模型名称。
4. 供应商类型。
5. 请求协议。
6. 上下文长度。
7. 是否支持流式输出。
8. 默认温度、Top P、最大输出 Token。
9. 任务用途，例如问答、摘要、翻译、Markdown 转换、复核辅助。

系统应优先支持 OpenAI-compatible API，以便兼容多种外部商业模型、自建模型网关和本地推理服务。

### 4.4 完整前后端与数据库模块

系统应至少包含：

1. 客户端前端模块。
2. 后端 API 模块。
3. 用户鉴权模块。
4. 数据库模块。
5. 文件存储模块。
6. 文档解析模块。
7. Markdown 转换模块。
8. 检索索引模块。
9. 向量数据库模块。
10. 模型调用模块。
11. 本地模型运行模块。
12. 异步任务队列模块。
13. 引用跳转模块。
14. 复核与批注模块。
15. 日志与审计模块。

### 4.5 Hugging Face 模型导入与运行

系统应支持导入 Hugging Face 上发布的大语言模型权重文件，也应支持导入用户本地微调后的模型目录。导入方式包括：

1. 输入 Hugging Face 模型 ID。
2. 输入模型仓库地址。
3. 上传本地模型文件。
4. 指定服务器上的模型目录。
5. 导入 LoRA Adapter 或其他微调权重。
6. 导入 GGUF 量化模型。

系统应检测模型格式、参数规模、上下文长度、推理后端要求和本机硬件条件。若硬件不足，应提示用户改用量化模型、服务器推理或外部 API。

### 4.6 文档上传与统一管理

系统应支持上传：

1. PDF。
2. DOCX。
3. Markdown。
4. TXT。
5. 后续可扩展 HTML、CSV、XLSX、EPUB、网页快照等格式。

所有文件应进入统一资料库，并支持元数据管理、标签管理、版本管理、权限管理和处理状态查看。

### 4.7 Markdown 统一中间格式

系统内部应以 Markdown 或类 Markdown 结构化文本作为统一中间格式。用户可以直接上传 Markdown 文件；也可以通过本地解析器或外部商业大模型 API 将 PDF、DOCX 等文件转换为 Markdown。

转换后的 Markdown 不应替代原始文件。系统必须保留原始文件，并维护 Markdown 段落与原始文件页码、段落、坐标或字符偏移之间的映射关系。

### 4.8 模型输出引用溯源

模型回答必须尽可能附带证据来源。每条关键结论应关联：

1. 原始文件。
2. 文件标题。
3. 页码或章节。
4. 段落或条文编号。
5. 引用文本。
6. 文档块 ID。
7. 跳转位置。
8. 置信度或引用质量标记。

前端应将模型输出正文、引用来源和复核状态清晰分离。

### 4.9 一键跳转原文

用户点击引用来源后，系统应跳转至对应文件的对应位置：

1. PDF：跳转页码，并尽可能高亮文本区域。
2. DOCX：跳转至转换后的结构化段落，后续可支持 DOCX 原文预览定位。
3. Markdown：跳转至标题锚点或段落。
4. TXT：跳转至段落或字符偏移。

这是系统区别于普通大模型问答工具的核心功能。

### 4.10 二次复核与人工校正

模型输出后，用户可以进行人工复核。复核状态包括：

1. 未复核。
2. 已复核，可信。
3. 已复核，引用不足。
4. 已复核，结论需修改。
5. 已复核，存在误译。
6. 已复核，存在幻觉。
7. 已复核，与原文无关。

系统应保存复核人、复核时间、复核意见、修订内容和相关引用。

### 4.11 快速翻译与解释

用户在阅读资料或复核输出时，可以选中文本并调用翻译、解释、摘要功能。系统应支持两类模型：

1. 外部商业大模型 API。
2. 用户导入或微调后的本地模型。

系统应明确显示当前使用的模型来源，避免用户混淆外部模型、本地模型和微调模型。

## 5. 前端设计要求

### 5.1 前端设计原则

前端开发不能简单拼装常见开源 UI 模板或直接套用成熟产品界面。系统应建立原创的模块化视觉设计体系，先形成一批具有一致风格的原始界面元素，再通过排列组合构成 Windows、Linux、Android 等多端应用界面。

前端设计应兼顾：

1. 原创性。
2. 模块化。
3. 一致性。
4. 可复用性。
5. 多端适配。
6. 可实现性。
7. 可形成清晰的界面设计稿、组件库和版本记录。

### 5.2 原始元素设计库

系统应先设计一批原创 UI 原始元素，而不是直接进入页面拼装。原始元素包括：

1. 应用 Logo 与品牌符号。
2. 主导航栏形态。
3. 侧边栏形态。
4. 顶部状态栏形态。
5. 工作区切换器。
6. 文件卡片。
7. 文档状态标签。
8. 模型状态指示器。
9. 引用来源卡片。
10. 复核状态徽章。
11. 对话气泡。
12. 文档块引用标识。
13. 原文跳转按钮。
14. 快速翻译浮层。
15. 批注气泡。
16. 检索结果卡片。
17. 模型选择器。
18. 任务模板按钮。
19. 文档目录树。
20. 多端底部操作栏。
21. 进度加载组件。
22. 空状态插画。
23. 错误状态插画。
24. 文件处理流程图标。
25. 引用可信度视觉标记。

这些元素应形成统一设计规范，包括尺寸、圆角、间距、阴影、描边、图标风格、状态变化、动效规则和响应式行为。

### 5.3 界面组合策略

页面不应作为一次性静态设计完成，而应由原创元素组合形成。推荐采用“元素 — 组件 — 模块 — 页面 — 多端布局”的五层设计体系：

第一层：视觉元素。例如按钮、图标、标签、徽章、输入框、分割线。

第二层：功能组件。例如文件卡片、引用卡片、模型选择器、复核状态控件。

第三层：业务模块。例如资料库面板、文档阅读面板、AI 回答面板、引用复核面板。

第四层：完整页面。例如工作台首页、资料库页、文档阅读页、模型问答页、复核中心页。

第五层：多端布局。例如桌面端三栏布局、移动端底部导航布局、平板端双栏布局。

### 5.4 关键页面的原创界面方向

#### 5.4.1 工作台首页

首页应体现系统的“可信知识工作”定位。建议采用信息仪表盘布局，包括资料库概况、最近处理文件、待复核输出、模型运行状态、快捷任务入口。

视觉重点应放在“资料 — 模型 — 引用 — 复核”的流程感，而不是普通聊天工具界面。

#### 5.4.2 资料库页面

资料库页面应突出文件组织能力。建议采用左侧筛选栏、中部文件卡片列表、右侧元数据预览的结构。文件卡片应显示文件类型、处理状态、语言、标签、引用次数和最后复核时间。

#### 5.4.3 文档阅读页面

文档阅读页面是外观设计重点。建议采用“原文阅读区 + 结构目录 + AI 辅助浮层 + 批注层”的组合。用户选中文本后，浮层出现翻译、解释、摘要、创建引用、添加批注等操作。

#### 5.4.4 模型问答页面

模型问答页不应只是普通聊天窗口。建议采用“三段式可信回答结构”：

1. 用户问题与任务设置。
2. 模型回答正文。
3. 引用证据与复核面板。

回答中的每个关键结论都可以对应一个引用标识，点击后在右侧打开引用证据卡片。

#### 5.4.5 复核中心页面

复核中心应突出“待处理事项”和“证据质量”。建议按未复核、引用不足、疑似幻觉、已确认可信等状态分类。每条复核任务应显示模型结论、引用原文、差异标记、复核按钮和处理记录。

#### 5.4.6 模型管理页面

模型管理页面应区分外部 API 模型、本地模型、微调模型和嵌入模型。每个模型以状态卡片展示，包括来源、用途、上下文长度、运行状态、硬件需求和调用权限。

### 5.5 多端界面适配

桌面端建议采用多栏布局：

1. 左侧主导航。
2. 中部主工作区。
3. 右侧引用、批注或模型信息面板。

Android 端建议采用层级式布局：

1. 底部导航切换主要模块。
2. 文档阅读和 AI 回答分屏可折叠。
3. 引用来源以底部抽屉展示。
4. 快速翻译和批注采用浮层交互。

Linux 与 Windows 桌面端应保持一致的布局结构，但允许系统标题栏、窗口控制和字体渲染根据平台适配。

### 5.6 设计资产管理

前端设计资产应进行版本化管理。建议保存：

1. 原始设计稿。
2. 组件库设计稿。
3. 页面组合设计稿。
4. 多端适配设计稿。
5. 设计迭代记录。
6. 图标源文件。
7. 动效说明。
8. 组件命名规范。
9. 页面截图版本。
10. 与代码实现对应的组件文档。

## 6. 系统架构设计

### 6.1 总体架构

系统采用前后端分离与模块化服务架构：

1. 客户端层：Windows、Linux、Android 应用。
2. 网关与 API 层：统一接口、鉴权、限流、请求转发。
3. 业务服务层：用户、工作区、资料库、对话、复核、批注。
4. 文档处理层：文件解析、OCR、Markdown 转换、文档分块、位置映射。
5. 检索层：全文检索、向量检索、混合检索、重排序。
6. 模型层：外部 API 调用、本地模型推理、嵌入模型、模型配置。
7. 数据层：关系型数据库、向量数据库、对象存储、缓存。
8. 任务层：异步文件处理、索引构建、批量转换、模型导入。
9. 审计层：调用日志、复核记录、引用质量记录、系统操作日志。

### 6.2 推荐技术栈

前端：Flutter。

后端：Python FastAPI。

数据库：PostgreSQL。

向量索引：pgvector。

对象存储：MinIO 或本地文件系统。

缓存与任务队列：Redis。

异步任务：Celery、RQ 或 Dramatiq。

文档解析：PyMuPDF、pdfplumber、python-docx、Pandoc、Tesseract OCR。

模型调用：OpenAI-compatible API 适配层、Transformers、llama.cpp、vLLM、Ollama 或 LM Studio 兼容服务。

部署：Docker Compose，后续可扩展 Kubernetes。

### 6.3 模块化原则

系统应避免强耦合。以下能力应抽象为可替换模块：

1. LLM Provider。

6. 文件存储。

## 7. 文档处理与可信引用机制

### 7.1 文档处理流水线

文件上传后进入异步处理流程：

1. 文件校验。
2. 病毒与风险扫描，视部署条件实现。
3. 原始文件入库。
4. 文本提取。
5. OCR 处理，若需要。
6. Markdown 转换。
7. 结构解析。
8. 原文位置映射。
9. 文档分块。
10. 向量化。
11. 全文索引。
12. 处理结果回写。

### 7.2 文档分块策略

文档分块应优先保留文档自然结构。系统不应只按固定字数机械切分，而应综合考虑：

1. 标题层级。
2. 条款编号。
3. 段落边界。
4. 表格结构。
5. 脚注。
6. 页码。
7. 语义完整性。
8. 上下文窗口。

每个文档块应有唯一 ID，并保存来源文件、页码、段落、标题路径和文本内容。

### 7.3 检索增强生成流程

RAG 流程如下：

1. 用户提出问题或任务。
2. 系统识别任务类型。
3. 系统根据资料范围和元数据过滤候选文档。
4. 执行全文检索与向量检索。
5. 进行混合排序和重排序。
6. 选出候选文档块。
7. 构造带 chunk_id 的上下文。
8. 调用模型生成回答。
9. 要求模型引用给定 chunk_id。
10. 后端解析引用并写入 citation 表。
11. 前端展示回答和证据来源。
12. 用户跳转原文复核。

### 7.4 幻觉控制策略

系统应从多个层面降低幻觉风险：

1. 提示词层：要求模型基于给定资料回答，不得编造来源。
2. 检索层：提高召回质量，避免错误上下文。
3. 引用层：所有关键结论必须关联文档块。
4. UI 层：对无引用回答进行显著提示。
5. 复核层：允许用户标记幻觉、误引和引用不足。
6. 日志层：保存问题、检索片段、模型回答、引用和复核结果。
7. 评估层：定期统计引用准确率、人工驳回率和疑似幻觉率。

## 9. 数据库设计草案

### 9.1 users 用户表

建议 users 用户表包含以下字段：

```text
id
email
password_hash
display_name
role
status
created_at
updated_at
last_login_at
```

其中，role 字段用于标记用户的全局角色。MVP 阶段可取值为：

```text
admin
user
```

admin 表示管理员，拥有系统全部管理权限。user 表示普通用户，只能使用模型、查看资料、执行标记、批注和复核等操作。

status 字段用于标记账号状态。可取值为：

```text
active
disabled
pending
```

active 表示账号正常可用。disabled 表示账号已被管理员禁用。pending 表示账号已创建但尚未完成激活或审核。

### 9.2 workspaces 工作区表

字段包括：id、name、description、owner_id、visibility、created_at、updated_at。

### 9.3 workspace_members 工作区成员表

建议 workspace_members 工作区成员表包含以下字段：

```text
id
workspace_id
user_id
role
joined_at
invited_by
status
```

其中，role 字段用于标记用户在具体工作区内的角色。MVP 阶段可取值为：

```text
admin
user
```

工作区内 admin 可以管理该工作区的模型配置、资料、成员和系统设置。工作区内 user 只能使用已配置模型、查看或上传资料、执行复核、标记和批注。

status 字段用于标记该用户在工作区中的成员状态。可取值为：

```text
active
removed
pending
```

active 表示该用户是当前有效成员。removed 表示该用户已被移出工作区。pending 表示邀请已发送但用户尚未接受。

### 9.4 documents 文件表

字段包括：id、workspace_id、title、original_filename、file_type、language、category、tags、original_file_path、markdown_file_path、processing_status、uploaded_by、created_at、updated_at。

### 9.5 document_metadata 文件元数据表

字段包括：id、document_id、metadata_key、metadata_value、created_at、updated_at。

该表用于支持不同行业的动态元数据。例如法律场景可以存储法域、发布机构、生效日期；企业知识库场景可以存储部门、项目、版本号；学术场景可以存储作者、期刊、年份。

### 9.6 document_chunks 文档块表

字段包括：id、document_id、chunk_index、heading_path、content、page_start、page_end、paragraph_index、char_start、char_end、bbox、embedding、created_at。

### 9.7 model_configs 模型配置表

字段包括：id、workspace_id、model_name、provider_type、base_url、encrypted_api_key、context_length、support_streaming、task_scope、created_by、created_at、updated_at。

### 9.8 local_models 本地模型表

字段包括：id、workspace_id、model_name、source、model_path、weight_format、parameter_size、quantization、context_length、supported_languages、supported_tasks、runtime_backend、status、created_at、updated_at。

### 9.9 conversations 对话表

字段包括：id、workspace_id、user_id、title、model_id、created_at、updated_at。

### 9.10 messages 消息表

字段包括：id、conversation_id、role、content、model_name、created_at。

### 9.11 citations 引用表

字段包括：id、message_id、document_id、chunk_id、quoted_text、page_number、paragraph_index、bbox、confidence、created_at。

### 9.12 review_records 复核记录表

字段包括：id、citation_id、reviewer_id、status、comment、corrected_text、created_at、updated_at。

### 9.13 annotations 批注表

字段包括：id、document_id、user_id、selected_text、comment、page_number、paragraph_index、bbox、created_at、updated_at。

### 9.14 ui_design_assets 前端设计资产表

字段包括：id、asset_name、asset_type、version、description、file_path、created_by、created_at、updated_at。

该表用于管理原创 UI 元素、组件、图标、页面设计稿和多端截图，便于后续版本追踪。

DELETE /api/models/configs/{config_id}：删除模型配置。

POST /api/annotations：创建批注。

## 18. 风险评估

### 18.1 开发范围过大风险

系统同时涉及多端客户端、后端、数据库、文档解析、RAG、本地模型、UI 设计系统和开源生态。初期如果同时推进所有功能，风险较高。

应对策略：优先完成“资料库 + 外部模型 API + 引用溯源 + 原文跳转”的核心闭环。

### 18.2 幻觉控制效果不稳定风险

RAG 并不能完全消除幻觉。检索错误、上下文不足、模型不服从指令都会导致错误回答。

应对策略：引入引用约束、无来源警告、人工复核、复核统计和提示词迭代。

### 18.3 文档解析准确性风险

复杂 PDF、扫描件、表格、脚注和多栏排版可能导致 Markdown 转换错误。

应对策略：保留原始文件、保存映射、允许人工修订、支持外部模型辅助转换。

### 18.4 本地模型运行风险

大模型对硬件要求高，普通设备可能无法运行。

应对策略：将本地模型作为增强功能，优先支持服务器推理、量化模型和外部 API。

### 18.5 前端原创性风险

如果直接套用成熟 UI 框架默认样式，可能削弱界面原创性和后续外观设计布局空间。

应对策略：先设计原创元素库，再进行页面组合；保留设计稿和迭代记录；避免直接复制竞品界面。

### 18.6 开源许可证风险

不同开源依赖许可证可能对后续使用、分发和商业化产生影响。

应对策略：建立依赖许可证清单，开源前进行许可证审查。
